# GitForge Philosophy

GitForge exists because repeated work should become a system.

Publishing a local project to GitHub is a useful workflow, but it involves repeated steps that are easy to forget:

- initialize Git
- stage files
- commit
- create the repository
- push
- set description
- add topics
- verify the result

GitForge turns that workflow into a repeatable tool.

## Rule

GitForge should stay focused on one job:

**Turn a local project folder into a polished GitHub repository.**

If a feature does not support that job, it belongs in another project.
